📐 โมเดลการโปรแกรม CUDA

โมเดลการโปรแกรม CUDA กำหนดวิธีที่ซอฟต์แวร์แสดงความขนานเพื่อรันบน GPU ของ NVIDIA เป็นโมเดล heterogeneous: โค้ดเริ่มต้นที่ CPU แล้วส่งงานไปยัง GPU

ระบบ Heterogeneous
🖥️ Host (CPU)
CPU และ DRAM ที่เชื่อมต่อโดยตรง (System Memory) — โค้ดเริ่มต้นที่นี่เสมอ เรียกใช้ CUDA API เพื่อส่งงานไป GPU
⚡ Device (GPU)
GPU และ DRAM ที่เชื่อมต่อโดยตรง (Device Memory / HBM) — รัน Kernel ที่ launch จาก Host
ลำดับชั้น Thread (Thread Hierarchy)
Grid ชุด Block ทั้งหมดในการ launch kernel ครั้งเดียว — ขนาดสูงสุด (2³¹−1) × 65535 × 65535
Thread Block Thread สูงสุด 1024 ตัว รันบน SM เดียวกัน — มี Shared Memory ร่วมกัน
Warp 32 threads — หน่วยการรันของ hardware รันคำสั่งเดียวกันพร้อมกัน (SIMT)
Thread หน่วยพื้นฐาน มี registers และ execution context เป็นของตัวเอง
🔮 Thread Block Clusters (Compute Capability 9.0+)
ระดับใหม่ระหว่าง Grid และ Block — กลุ่ม Block ที่อยู่บน SM ใน GPC เดียวกัน
เปิดใช้งาน Distributed Shared Memory: threads สามารถเข้าถึง shared memory ของ Block อื่นใน Cluster ได้ผ่าน cluster.map_shared_rank()
ต้องใช้ cluster.sync() สำหรับ synchronization ระดับ cluster
Syntax การ Launch Kernel
CUDA C++
// ไวยากรณ์ triple-chevron: <<<grid_dims, block_dims, shared_mem_bytes, stream>>>
myKernel<<<gridDim, blockDim, sharedMemBytes, stream>>>(args...);

// ตัวอย่างจริง: GEMM kernel
dim3 grid(M/TILE, N/TILE);          // กี่ block ใน grid
dim3 block(TILE, TILE);              // กี่ thread ใน block
gemmKernel<<<grid, block, 0, stream>>>(A, B, C, M, N, K);
Built-in Variables ภายใน Kernel
Variableความหมายตัวอย่างการใช้
threadIdx.{x,y,z}ตำแหน่ง thread ภายใน blockระบุว่า thread นี้ทำงานกับ data ชิ้นไหน
blockIdx.{x,y,z}ตำแหน่ง block ภายใน gridระบุว่า block นี้อยู่ส่วนไหนของ matrix
blockDim.{x,y,z}ขนาดของ blockใช้คำนวณ global index
gridDim.{x,y,z}ขนาดของ gridใช้ใน stride loop
warpSizeขนาด warp (ค่าคงที่ = 32)ใช้ใน warp-level primitives
✅ กฎสำคัญ: Block อิสระจากกัน
Thread block ต้องรันได้โดยไม่ขึ้นกับ block อื่น — ไม่มีลำดับการรันที่รับประกัน ไม่มี cross-block synchronization (ยกเว้น Cooperative Groups)
นี่คือเหตุผลที่ CUDA scaling จาก GPU 1 SM ถึง 1000 SM ด้วย binary เดียวกัน
ตัวอย่าง: Vector Addition
CUDA C++
__global__ void vecAdd(float* A, float* B, float* C, int n) {
    // คำนวณ index ของ thread นี้ใน array
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < n) {
        C[idx] = A[idx] + B[idx];
    }
}

// เรียกใช้จาก host:
int blockSize = 256;
int gridSize  = (n + blockSize - 1) / blockSize;  // ceiling division
vecAdd<<<gridSize, blockSize>>>(d_A, d_B, d_C, n);

⚙️ โมเดลการรัน CUDA (SIMT)

โมเดลการรัน CUDA กำหนดว่า thread ถูก schedule, รัน, และ synchronize กันอย่างไรบน GPU hardware — หน่วยฮาร์ดแวร์หลักคือ warp ซึ่งเป็น 32 threads ที่รัน lockstep ภายใต้ SIMT

SIMT คืออะไร?
SIMT (Single-Instruction Multiple-Thread)
  • Thread 32 ตัวใน warp รัน คำสั่งเดียวกัน พร้อมกัน
  • แต่ละ thread มี program counter, register, call stack เป็นของตัวเอง (CC 7.0+)
  • Thread สามารถแยก branch ได้ แต่จะถูก serialize
SIMD vs SIMT
  • SIMD: expose ความกว้างของ vector ให้ software — programmer จัดการ lane เอง
  • SIMT: programmer เขียนโค้ด per-thread แบบ scalar — hardware รวมเป็น SIMT เอง
Warp Scheduling
🔄 Zero-Cost Context Switch
SM เก็บ context ทั้งหมดของทุก warp ที่ active ไว้บน chip ตลอดเวลา
การสลับ warp ใช้เวลา 0 cycle — นี่คือวิธีที่ GPU ซ่อน memory latency:
ขณะที่ warp A รอ global memory load (400–800 cycle) → warp scheduler รัน warp B, C, D แทน
Warp Divergence
ตัวอย่าง — Divergence ที่ควรหลีกเลี่ยง
// ❌ BAD: ครึ่ง warp idle ในแต่ละ branch
if (threadIdx.x % 2 == 0) {
    doHeavyWork_A();   // thread 0, 2, 4... active; thread 1, 3, 5... masked
} else {
    doHeavyWork_B();   // thread 1, 3, 5... active; thread 0, 2, 4... masked
}
// ทำให้ใช้เวลา 2x เมื่อเทียบกับ path เดียว

// ✅ GOOD: ใช้ threadIdx.x ให้ threads ใน warp เดียวกันไป path เดียวกัน
if (blockIdx.x % 2 == 0) {
    doHeavyWork_A();   // ทุก thread ใน block ไป path เดียวกัน — ไม่มี divergence
} else {
    doHeavyWork_B();
}
Independent Thread Scheduling (CC 7.0+)
ก่อน CC 7.0 (Kepler, Maxwell, Pascal)
PC เดียวสำหรับทั้ง warp — threads ใน divergent region ไม่สามารถส่งสัญญาณหากัน อาจ deadlock ได้
CC 7.0+ (Volta, Turing, Ampere, Hopper, Blackwell)
Independent Thread Scheduling — แต่ละ thread มี PC และ call stack เป็นของตัวเอง
SM สามารถ yield thread ได้ระดับ per-thread
Schedule optimizer จัดกลุ่ม active threads เป็น SIMT units เพื่อ throughput
⚠️ Compatibility Warning
โค้ดที่เขียนโดยสมมติว่า threads ใน warp รัน lockstep ทุก instruction (warp-synchronous code) อาจทำงานผิดพลาดบน CC 7.0+
ต้องใช้ __syncwarp() อย่างชัดเจนสำหรับการ sync ภายใน warp
Thread Scopes
ScopePTXThreads ที่เห็น operationจุด coherency
thread_scope_thread—thread ตัวเองเท่านั้น—
thread_scope_block.ctaทุก thread ใน block เดียวกันL1

Occupancy Tradeoff: Shared Mem vs L1
⚖️ Shared Memory ↔ L1 คือ Zero-Sum
Shared memory และ L1 cache แชร์ physical space (Unified Data Cache) เดียวกันบน SM
การใช้ shared memory มาก → L1 น้อยลง → อาจมี cache miss มากขึ้น
ใช้ cudaFuncSetAttribute(kernel, cudaFuncAttributePreferredSharedMemoryCarveout, pct) เพื่อปรับสัดส่วน
Register Spilling
🚨 Register Spilling คือหายนะ
เมื่อ kernel ใช้ register มากกว่าที่ SM มี → compiler spill register ไปยัง Local Memory
Local Memory อยู่ใน Global DRAM จริง ๆ (ไม่ใช่ on-chip) → latency เหมือน Global Memory (400–800 cycle)
ป้องกันด้วย: __launch_bounds__(maxThreads, minBlocks) หรือ --maxrregcount

🔗 Memory Coalescing

Memory coalescing คือการปรับแต่งที่ให้ผลมากที่สุดสำหรับ kernel ที่ถูก global-memory bound เมื่อ threads ใน warp เข้าถึง memory พร้อมกัน hardware สามารถรวม request เป็น transaction เดียวได้ — แต่ต้องเป็นที่อยู่ที่ต่อเนื่องกันเท่านั้น

กฎพื้นฐาน
✅ Coalesced (ดี)
32 threads เข้าถึง 32 ที่อยู่ที่ต่อเนื่องกัน → hardware รวมเป็น 1 transaction (128 bytes)
A[threadIdx.x] // thread 0→A[0], thread 1→A[1]...
Bandwidth: ~3.35 TB/s (peak)
❌ Uncoalesced (แย่)
32 threads เข้าถึง 32 ที่อยู่แบบสุ่ม → hardware ต้องทำ 32 transactions แยกกัน
A[threadIdx.x * 32] // stride ใหญ่เกินไป
Bandwidth: ~105 GB/s (ลดลง 32x!)
ตัวอย่าง Pattern การเข้าถึง
Patternผลลัพธ์เหตุผล
A[threadIdx.x]✅ Coalescedที่อยู่ต่อเนื่องกัน
A[threadIdx.x * 2]⚠️ บางส่วนstride 2 — ประสิทธิภาพ 50%
A[threadIdx.x * 32]❌ Uncoalescedstride ใหญ่ — 32 transactions
column access ใน row-major matrix❌ Uncoalescedstride = N (ขนาด row)
A[index[threadIdx.x]] แบบสุ่ม❌ Uncoalescedgather pattern
ผ่าน Shared Memory staging✅ Coalescedload ผ่าน shared memory ก่อน
กรณีศึกษา: Matrix Transpose
Matrix Transpose — ใช้ Shared Memory แก้ Uncoalesced Write
// Shared memory padding เพื่อแก้ทั้ง bank conflict และ uncoalesced write
__global__ void smem_transpose(int m, float *a, float *c) {
    // +1 คือ padding เพื่อหลีกเลี่ยง bank conflict (ดูหน้า Shared Memory)
    __shared__ float smem[32][33];

    int tileCol = blockDim.x * blockIdx.x;
    int tileRow = blockDim.y * blockIdx.y;

    // READ จาก global: threadIdx.x เปลี่ยนเร็ว → Coalesced ✅
    smem[threadIdx.x][threadIdx.y] = a[(tileRow + threadIdx.y)*m + (tileCol + threadIdx.x)];
    __syncthreads();

    // WRITE ไป global: threadIdx.x เปลี่ยนเร็ว → Coalesced ✅
    c[(tileCol + threadIdx.y)*m + (tileRow + threadIdx.x)] = smem[threadIdx.y][threadIdx.x];
}

🧩 Shared Memory & Tiling

Shared Memory คือ on-chip SRAM บน SM — latency ~20–40 cycle เทียบกับ Global Memory 400–800 cycle Pattern Tiling คือเทคนิคหลักที่ใช้ประโยชน์จาก shared memory: โหลด data จาก global มาที่ shared memory ครั้งเดียว แล้วนำมาใช้ซ้ำหลายครั้ง

Pattern Tiling (แม่แบบหลัก)
Pattern พื้นฐาน
for (int tile = 0; tile < K/TILE; tile++) {
    // 1. โหลด tile จาก global → shared memory (cooperative: แต่ละ thread โหลด 1 element)
    smem[threadIdx.y][threadIdx.x] = A[row * K + tile * TILE + threadIdx.x];
    smem2[threadIdx.y][threadIdx.x] = B[(tile * TILE + threadIdx.y) * N + col];

    // 2. รอให้ทุก thread โหลดเสร็จก่อน (สำคัญมาก!)
    __syncthreads();

    // 3. คำนวณโดยใช้ data ใน shared memory (เร็วกว่า global 10–20x)
    for (int k = 0; k < TILE; k++) {
        sum += smem[threadIdx.y][k] * smem2[k][threadIdx.x];
    }

    // 4. รอให้ compute เสร็จก่อน tile ถัดไปจะ overwrite shared mem
    __syncthreads();
}
ทำไม Tiling ถึงช่วย?
❌ Naive (ไม่มี Tiling)
สำหรับ GEMM M×K × K×N: แต่ละ thread อ่าน K elements จาก A และ K จาก B
รวม global reads: 2MNK
✅ Tiled (Tile size = T)
แต่ละ block โหลด tile T×T ครั้งเดียวสำหรับทุกการคำนวณใน tile
รวม global reads: 2MNK/T → ลดลง T เท่า!
สำหรับ T=32: ประสิทธิภาพเพิ่มขึ้น 32 เท่า
Bank Conflicts

Shared memory แบ่งเป็น 32 banks (interleaved ทุก 4 bytes) หาก threads ใน warp เดียวกันเข้าถึง bank เดียวกันพร้อมกัน → serialized (penalty)

Pattern การเข้าถึงผลลัพธ์เหตุผล
smem[threadIdx.x]✅ ไม่มี conflictthread 0→bank0, thread 1→bank1...
smem[threadIdx.x * 2]⚠️ 2-way conflict16 คู่แชร์ bank กัน
smem[threadIdx.x * 32]❌ 32-way conflictทุก thread อยู่ bank 0
ทุก thread อ่าน smem[0]✅ Broadcasthardware broadcast → ไม่ conflict
แก้ Bank Conflict ด้วย Padding
เทคนิค +1 Padding
// ❌ มี bank conflict เมื่อเข้าถึง column
__shared__ float smem[32][32];

// ✅ เพิ่ม 1 column เพื่อ shift bank alignment
__shared__ float smem[32][33];  // +1 padding
// ตอนนี้การเข้าถึง column ไม่ conflict แล้ว เพราะแต่ละ row shift ไป 4 bytes
การ Allocate Shared Memory
Static vs Dynamic Allocation
// Static: ขนาดกำหนดตอน compile
__global__ void kernel() {
    __shared__ float smem[1024];   // 4 KB
    // ...
}

// Dynamic: กำหนดตอน launch (flexible แต่ต้องระวัง alignment)
__global__ void kernel() {
    extern __shared__ float smem[];   // ขนาดจาก arg ที่ 3 ของ <<<>>>
    // ...
}
// การเรียก:
kernel<<<grid, block, 1024 * sizeof(float)>>>();

🌊 CUDA Streams & Events

Streams และ Events เป็น abstraction หลักสำหรับการรันแบบ asynchronous concurrent — ช่วยให้ทับซ้อน: host computation, device computation, memory transfers H→D, D→H, และ peer-to-peer ได้พร้อมกัน

Stream คืออะไร?
🌊 Stream = Work Queue แบบ In-Order
operations ใน stream เดียวกันรัน ตามลำดับที่ enqueue
operations ใน stream ต่างกันรันได้ พร้อมกัน (ขึ้นกับ hardware resources)
Kernel launch และ memory copy ใน stream return ทันที (asynchronous กับ host)
การสร้างและใช้งาน Stream
cudaStream_t stream1, stream2;
cudaStreamCreate(&stream1);
cudaStreamCreate(&stream2);

// Ping-Pong pattern: ทำงาน 2 stream ทับซ้อนกัน
cudaMemcpyAsync(d_A, h_A, size, cudaMemcpyHostToDevice, stream1);  // copy chunk A
cudaMemcpyAsync(d_B, h_B, size, cudaMemcpyHostToDevice, stream2);  // copy chunk B พร้อมกัน!

processKernel<<<grid, block, 0, stream1>>>(d_A, d_out1);  // process A
processKernel<<<grid, block, 0, stream2>>>(d_B, d_out2);  // process B พร้อมกัน!

cudaStreamSynchronize(stream1);
cudaStreamSynchronize(stream2);

cudaStreamDestroy(stream1);
cudaStreamDestroy(stream2);
Default Stream (NULL Stream)
ประเภท Streamสร้างด้วยbehavior กับ NULL stream

Non-blockingcudaStreamCreateWithFlags(s, cudaStreamNonBlocking)อิสระจาก NULL stream — แนะนำสำหรับ production
Per-thread defaultflag --default-stream per-threadแต่ละ host thread มี default stream ของตัวเอง
Events — สำหรับ Timing และ Cross-Stream Sync
การวัดเวลา Kernel ด้วย Events
cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start, stream);         // บันทึกเวลาเริ่ม
myKernel<<<grid, block, 0, stream>>>(args);
cudaEventRecord(stop, stream);          // บันทึกเวลาหยุด

cudaStreamSynchronize(stream);          // รอให้ stop event เสร็จ

float ms;
cudaEventElapsedTime(&ms, start, stop); // คำนวณเวลาเป็น millisecond
printf("Kernel time: %.3f ms\n", ms);

cudaEventDestroy(start);
cudaEventDestroy(stop);
Cross-Stream Dependency ด้วย Events
// บังคับให้ stream2 รอจน stream1 ถึง event
cudaEventRecord(event, stream1);
cudaStreamWaitEvent(stream2, event, 0);
// operations ที่ enqueue stream2 หลังจากนี้จะรอ event ใน stream1 ก่อน
Pinned Memory สำหรับ Async Copy
⚠️ Host Memory ต้องเป็น Pinned สำหรับ Async จริง

(trade-off: pinned memory ลด available RAM สำหรับ OS)

📊 Occupancy Optimization

Occupancy = (active warps per SM) ÷ (maximum resident warps per SM) — การมี occupancy สูงช่วยซ่อน memory latency: ขณะที่ warp หนึ่งรอ memory อีก warp จะรันแทน

⏳ Async Barriers & Pipelines

Hardware-accelerated async primitives ที่นำมาใน Ampere (CC 8.0) และขยายใน Hopper (CC 9.0) ช่วยให้การเคลื่อนย้าย data ระหว่าง global และ shared memory ดำเนินการได้อิสระจากการรัน thread — เปิดใช้งาน compute/memory overlap ภายใน kernel เดียว

ปัญหา: Synchronous Copy ทำให้ Thread หยุดรอ
Pattern เดิม — thread หยุดรอตลอด copy phase
// ❌ แบบเดิม: ทุก thread block ที่ __syncthreads()
for (int batch = 0; batch < n; batch++) {
    shared[tid] = global[bid + tid];     // copy ผ่าน register (2 ขั้น)
    __syncthreads();                      // thread ทั้งหมดรอ copy เสร็จ
    compute(global_out, shared);          // คำนวณ
    __syncthreads();
}

// ✅ Async: copy และ compute ทับซ้อนกันได้
for (int batch = 0; batch < n; batch++) {
    cg::memcpy_async(block, shared, global + offset, block.size());  // เริ่ม async copy
    // ทำงานอิสระได้ระหว่างนี้...
    cg::wait(block);                      // รอเฉพาะตอนต้องใช้ data
    compute(global_out, shared);
    block.sync();
}
cuda::barrier — Arrive/Wait Pattern
💡 แนวคิดหลัก: แยก "แจ้ง" ออกจาก "รอ"
  • bar.arrive() — thread แจ้งว่าทำ phase นี้เสร็จแล้ว (ไม่ block!)
  • ระหว่าง arrive กับ wait: thread ทำงานอื่นที่ไม่ขึ้นกับ data ได้
  • bar.wait(token) — block จนกว่าทุก thread arrive ครบ

Double Buffering ด้วย cuda::pipeline
Double Buffering — ทับซ้อน Load Stage N+1 กับ Compute Stage N
#include <cuda/pipeline>

__global__ void double_buffered(T* in, T* out, int n) {
    constexpr int STAGES = 2;
    __shared__ T buf[STAGES][BLOCK];
    auto pipe = cuda::make_pipeline();

    // โหลด stage แรก
    pipe.producer_acquire();
    cuda::memcpy_async(buf[0], in, sizeof(T)*BLOCK, pipe);
    pipe.producer_commit();

    for (int i = 0; i < n; i++) {
        // โหลด stage ถัดไป ขณะที่คำนวณ stage ปัจจุบัน
        pipe.producer_acquire();
        if (i+1 < n) cuda::memcpy_async(buf[(i+1)%2], in+(i+1)*BLOCK, sizeof(T)*BLOCK, pipe);
        pipe.producer_commit();

        pipe.consumer_wait();              // รอ stage i เสร็จ
        compute(out + i*BLOCK, buf[i%2]); // คำนวณด้วย data stage i
        pipe.consumer_release();
    }
}

🖥️ Multi-GPU Programming

Multi-GPU programming ช่วยแก้ปัญหาที่ใหญ่เกินกว่า GPU เดียวจะรองรับ CUDA ให้ host API, driver infrastructure, และ hardware technology สำหรับการประสานงานระหว่าง GPU

การจัดการ Device
Device Enumeration และ Selection
int deviceCount;
cudaGetDeviceCount(&deviceCount);

for (int dev = 0; dev < deviceCount; dev++) {
    cudaDeviceProp prop;
    cudaGetDeviceProperties(&prop, dev);
    printf("GPU %d: %s (CC %d.%d, %d SMs, %.1f GB)\n",
           dev, prop.name, prop.major, prop.minor,
           prop.multiProcessorCount,
           prop.totalGlobalMem / 1e9);
}

// เลือก GPU ที่ต้องการทำงานด้วย
cudaSetDevice(0);
float* d_ptr;
cudaMalloc(&d_ptr, size);   // allocate บน GPU 0
MyKernel<<<grid, block>>>(d_ptr);  // รันบน GPU 0
Peer-to-Peer Memory Transfer
P2P Copy และ Access
// ตรวจสอบว่า P2P ทำได้หรือไม่
int canAccess;
cudaDeviceCanAccessPeer(&canAccess, 0, 1);

if (canAccess) {
    // Copy โดยตรงระหว่าง GPU (ผ่าน NVLink หรือ PCIe)
    cudaMemcpyPeer(d_gpu1, 1, d_gpu0, 0, size);

    // หรือ enable peer access ให้ kernel บน GPU 1 อ่าน/เขียน memory GPU 0 ได้
    cudaSetDevice(1);
    cudaDeviceEnablePeerAccess(0, 0);
    MyKernel<<<grid, block>>>(ptr_on_gpu0);  // kernel GPU 1 เข้าถึง GPU 0's memory
}
